# Domain Events

Gopernicus ships a domain event bus that decouples use cases from side effects.
Cases emit events; bridge-layer subscribers react to them.  The bus abstraction
supports in-memory, Redis Streams, and a transactional outbox pattern.

## The `Bus` interface

Defined in `infrastructure/events/events.go`:

```go
type Bus interface {
    Emit(ctx context.Context, event Event, opts ...EmitOption) error
    Subscribe(topic string, handler Handler) (Subscription, error)
    Close(ctx context.Context) error
}
```

- **Emit** publishes an event.  By default events are dispatched
  asynchronously; use `events.WithSync()` to block until all handlers return.
- **Subscribe** registers a `Handler` for the given topic string.  Use `"*"`
  as a wildcard to receive all event types.  Returns a `Subscription` whose
  `Unsubscribe()` method removes the handler.
- **Close** shuts the bus down gracefully, draining pending async events up to
  the context deadline.

A `NoopBus` (`events.NewNoopBus()`) is provided for tests or when events are
disabled.

## `Event` interface and `BaseEvent`

Every domain event must implement the `Event` interface:

```go
type Event interface {
    Type() string
    OccurredAt() time.Time
    CorrelationID() string
}
```

`BaseEvent` is the standard building block.  Embed it and override `Type()`:

```go
type InvitationSentEvent struct {
    events.BaseEvent
    InvitationID string `json:"invitation_id"`
    Identifier   string `json:"identifier"`
    Token        string `json:"token"`
}

func (e InvitationSentEvent) Type() string { return "invitation.sent" }
```

`BaseEvent` provides:

- `EventType`, `Occurred`, `Correlation` fields (auto-populated by
  `NewBaseEvent`).
- Optional metadata: `Tenant`, `AggType`, `AggID` via chaining methods
  (`WithTenant`, `WithAggregate`).
- Outbox routing via `.ToOutbox()`.

Correlation IDs are generated by `events.GenerateID` (16 bytes crypto/rand,
hex-encoded).  Use `NewBaseEventWithCorrelation` to propagate an existing
correlation ID across related events.

## Defining event types

Convention is lowercase, dot-separated: `"invitation.sent"`, `"member.added"`,
`"email.verified"`.  Define events in the use-case package where they
originate (e.g., `core/auth/invitations/events.go`).

## Emitting from use cases

Cases call `bus.Emit(ctx, event)` after the core operation succeeds:

```go
_ = r.bus.Emit(ctx, InvitationSentEvent{
    BaseEvent:    events.NewBaseEvent("invitation.sent"),
    InvitationID: inv.InvitationID,
    Identifier:   input.Identifier,
    Token:        plainToken,
    InvitedBy:    input.InvitedBy,
})
```

Emit errors are typically logged but not propagated (fire-and-forget for async
events).

## Subscribing in bridges

Bridge-layer `Subscribers` structs register handlers on the bus during
application startup.  The pattern:

```go
type Subscribers struct {
    invitations *invitationscore.Inviter
    log         *slog.Logger
    subs        []events.Subscription
}

func (s *Subscribers) Register(bus events.Bus) error {
    sub, err := bus.Subscribe("email.verified",
        events.TypedHandler(s.handleEmailVerified))
    if err != nil {
        return err
    }
    s.subs = append(s.subs, sub)
    return nil
}
```

`Unsubscribe()` tears down all subscriptions at shutdown.

## `TypedHandler` for type-safe subscription

`events.TypedHandler[T]` wraps a typed function as a generic `Handler`.
Events that do not match type `T` are silently skipped:

```go
handler := events.TypedHandler(func(ctx context.Context, e UserCreatedEvent) error {
    return sendWelcomeEmail(e.Email)
})
bus.Subscribe("user.created", handler)
```

## Bus implementations

### Memory bus (`memorybus`)

In-process bus with configurable worker pool (default 4 goroutines) and
buffered async queue (default 1000).  Async events use
`context.WithoutCancel` to survive request cancellation.  Panics in handlers
are recovered and logged.  Suitable for single-instance and development.

### Redis Streams bus (`goredisbus`)

Uses XADD/XREADGROUP/XACK for durable, at-least-once delivery across
multiple process instances sharing the same consumer group.  Configuration
is via `Options` struct populated from environment variables
(`EVENT_BUS_STREAM_PREFIX`, `EVENT_BUS_CONSUMER_GROUP`,
`EVENT_BUS_WORKERS`, `EVENT_BUS_BLOCK_TIMEOUT`).

### Outbox pattern (`outbox`)

The `outbox.Bus` is a **decorator** around any other bus.  Events marked with
`.ToOutbox()` are intercepted and written to a Postgres `event_outbox` table
via the `OutboxWriter` interface.  Non-outbox events pass through to the
underlying bus.

The outbox is for critical business events requiring durable delivery with
retry.  A background worker stages, processes, and marks events as completed
with exponential backoff (`Options.RetryBaseDelay`, `Options.RetryMaxDelay`,
`Options.MaxRetries`).

Outbox events and bus events serve different purposes and are intentionally
separate -- switching the bus provider (memory to Redis Streams) does not
affect outbox routing.

## Event registry

`events.EventRegistry` is a handler router for outbox/queue workers.  It
supports exact match, prefix patterns (`"email.*"`), and wildcards (`"*"`).
Priority: exact > longest prefix > wildcard.
