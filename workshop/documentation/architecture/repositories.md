# Repository Pattern

How data access is structured in gopernicus: generation from SQL, layered interfaces, caching, and domain-level composition.

## Overview

Repositories are the data-access layer in gopernicus. They are generated from annotated `queries.sql` files using `gopernicus generate`. The generator produces a layered architecture:

```
Repository  (business logic: ID generation, pagination, event emission)
    |
CacheStore  (optional read caching + write invalidation)
    |
Store       (pgx-backed SQL execution)
```

Each layer implements the same `Storer` interface, which follows the hexagonal architecture principle of "accept interfaces, return structs."

## Generation from queries.sql

Every repository starts with a `queries.sql` file in the `gopernicus-cli/internal/frameworkrepos/` tree. The SQL file uses annotation comments to declare functions, filters, ordering, and search. In v2, `queries.sql` contains only **data annotations** -- protocol and auth annotations have moved to `bridge.yml`:

```sql
-- @func: List
-- @filter:conditions *
-- @search: ilike(resource_type, resource_id, relation, identifier)
-- @order: *
-- @max: 100
SELECT *
FROM invitations
WHERE $conditions AND $search
ORDER BY $order
LIMIT $limit
;

-- @func: Get
SELECT *
FROM invitations
WHERE invitation_id = @invitation_id
;

-- @func: Create
-- @fields: *,-created_at,-updated_at
INSERT INTO invitations ($fields) VALUES ($values) RETURNING *;

-- @func: Update
-- @fields: *,-invitation_id,-created_at
UPDATE invitations SET $fields WHERE invitation_id = @invitation_id RETURNING *;

-- @func: Delete
DELETE FROM invitations WHERE invitation_id = @invitation_id;
```

Key data annotations:

| Annotation | Purpose |
|---|---|
| `@func: Name` | Declares a store method |
| `@database` | Specifies which database from gopernicus.yml |
| `@filter:conditions *` | Generates a `FilterName` struct with all columns (or a subset) |
| `@search: ilike(col1, col2)` | Adds `SearchTerm` to the filter struct and generates ILIKE conditions |
| `@order: *` | Enables cursor-based pagination with OrderBy constants |
| `@max: 100` | Sets the maximum page limit constant |
| `@fields: *,-col` | Controls which columns appear in Create/Update input structs |
| `@cache` | Enables caching for a read query |
| `@event` | Declares a domain event to emit |
| `@scan` | Controls row scanning behavior |
| `@returns` | Overrides the return type |
| `@type` | Specifies type hints |

Protocol and auth annotations (`@http:json`, `@authenticated`, `@authorize`, `@auth.create`, `@with:permissions`, `@auth.relation`, `@auth.permission`) now live in the bridge layer's `bridge.yml` file.

Running `gopernicus generate` reads these SQL files and produces the generated code in the output repository package.

## Directory Layout

A generated repository for entity `invitations` under domain `rebac` produces:

```
core/repositories/rebac/invitations/
    generated.go          # DO NOT EDIT - entity types (Invitation, CreateInvitation, etc.),
                          # errors, Storer interface methods, Repository methods,
                          # OrderByFields, cursor helpers
    generated_cache.go    # DO NOT EDIT - CacheStore with read caching and write invalidation
    repository.go         # Bootstrap - Storer interface (with gopernicus:start/end markers),
                          # Repository struct, NewRepository
    fop.go                # Bootstrap - ordering defaults (DefaultOrderBy, DefaultLimit)
    cache.go              # Bootstrap - custom cache methods
    queries.sql           # Copied from frameworkrepos for reference

    invitationspgx/
        generated.go      # DO NOT EDIT - Store struct with pgx SQL methods
        store.go          # Bootstrap - custom store methods
```

Note: there is no `model.go` file. Entity types (`Invitation`, `CreateInvitation`, `UpdateInvitation`) are defined directly in `generated.go` with their final names.

## Generated vs Bootstrap Files

**Generated files** (`generated.go`, `generated_cache.go`, `invitationspgx/generated.go`) are overwritten on every `gopernicus generate` run. Never edit them directly. They carry the header:

```go
// Code generated by gopernicus. DO NOT EDIT.
```

**Bootstrap files** (`repository.go`, `fop.go`, `cache.go`, `invitationspgx/store.go`) are created once and never overwritten. They carry the header:

```go
// This file is created once by gopernicus and will NOT be overwritten.
```

You extend behavior by editing these bootstrap files.

## Key Generated Types

The generator produces these types directly in `generated.go` with their final names (no `Generated` prefix):

- **`Invitation`** -- the full database entity struct with `json` and `db` tags. Defined directly in generated.go.
- **`CreateInvitation`** -- input struct for inserts. Fields controlled by `@fields` annotation.
- **`UpdateInvitation`** -- input struct for updates. All fields are pointers (nil = unchanged).
- **`FilterList`**, **`FilterListByResource`**, etc. -- filter structs per list method with typed fields. Time fields get automatic `After`/`Before` range filters. All include `AuthorizedIDs []string` for prefilter authorization.
- **`OrderByFields`** -- map of orderable column names to their SQL representations.
- **`MaxLimit*` constants** -- per-list-method page size caps from `@max` annotations.
- **Sentinel errors** -- `ErrInvitationNotFound`, `ErrInvitationAlreadyExists`, `ErrInvitationInvalidReference`, `ErrInvitationInvalidInput`, all wrapping `sdk/errs` sentinels.

## The Storer Interface Pattern

The `Storer` interface lives in `repository.go` (developer-owned) with `// gopernicus:start` / `// gopernicus:end` markers. Generated methods are placed between the markers; custom methods go above:

```go
type Storer interface {
    // Custom methods go above the markers:
    // MyCustomQuery(ctx context.Context, tenantID string) ([]Invitation, error)

    // gopernicus:start
    List(ctx context.Context, filter FilterList, orderBy fop.Order, page fop.PageStringCursor, forPrevious bool) ([]Invitation, error)
    Get(ctx context.Context, invitationID string) (Invitation, error)
    Create(ctx context.Context, input CreateInvitation) (Invitation, error)
    Update(ctx context.Context, invitationID string, input UpdateInvitation) (Invitation, error)
    Delete(ctx context.Context, invitationID string) error
    GetByToken(ctx context.Context, tokenHash string, now time.Time) (Invitation, error)
    // gopernicus:end
}
```

To add a custom query, add the method signature above the `// gopernicus:start` marker in `repository.go`, then implement it in `invitationspgx/store.go`. The `CacheStore` wraps `Storer`, so custom methods pass through to the underlying store automatically.

## The Store Layer (pgx)

The generated `Store` in `invitationspgx/generated.go` implements `Storer` using PostgreSQL via pgx. It:

- Uses `pgx.NamedArgs` for parameterized queries
- Maps database errors (duplicate key, FK violation, check violation) to domain sentinel errors
- Builds dynamic `WHERE` clauses from filter structs
- Applies cursor-based pagination via `pgxdb.ApplyCursorPaginationFromToken`
- Accepts `pgxdb.Querier`, which is satisfied by both `*pgxpool.Pool` and `pgx.Tx`, enabling transaction composition

The bootstrap file `store.go` is where you write custom store methods that need raw SQL.

## The Cache Layer

`CacheStore` in `generated_cache.go` wraps a `Storer` with:

- **Read caching** for methods annotated with `@cache` (check-then-store pattern with configurable TTLs)
- **Write invalidation** that clears cached entries by primary key after mutations
- **Passthrough** when the cache instance is nil (no-op in development or tests)

The `CacheConfig` struct holds per-method TTL settings and a key prefix (e.g., `"rebac:invitations"`). Custom invalidation logic goes in the bootstrap `cache.go`.

## The Repository Layer

The `Repository` struct is defined in `repository.go` (bootstrapped) with its fields directly -- no embedding of a generated type:

```go
type Repository struct {
    store      Storer
    generateID func() string
    bus        events.Bus
}
```

Generated methods (List, Get, Create, etc.) are defined on `*Repository` in `generated.go`. These methods add:

- **ID generation** via `cryptids.GenerateID` (overridable)
- **Cursor pagination** for list methods (over-fetch by one to detect next page, previous-page detection)
- **Event emission** for create/update/delete when configured with an event bus

The `NewRepository` constructor lives in `repository.go` and is fully customizable:

```go
func NewRepository(store Storer, opts ...Option) *Repository {
    r := &Repository{
        store:      store,
        generateID: cryptids.GenerateID,
    }
    for _, opt := range opts {
        opt(r)
    }
    return r
}
```

## The Composite Pattern

Each domain (e.g., `rebac`) has a composite package that wires all its repositories together:

```go
// generated_composite.go (DO NOT EDIT)
type Repositories struct {
    Group                     *groups.Repository
    Invitation                *invitations.Repository
    RebacRelationship         *rebacrelationships.Repository
    RebacRelationshipMetadata *rebacrelationshipmetadata.Repository
}

func NewRepositories(log *slog.Logger, db pgxdb.Querier, c *cache.Cache, bus events.Bus) *Repositories {
    return &Repositories{
        Invitation: invitations.NewRepository(
            invitations.NewCacheStore(invitationspgx.NewStore(log, db), c),
            invitations.WithEventBus(bus),
        ),
        // ...
    }
}
```

The `Repositories` struct is defined directly in `generated_composite.go` with its final name -- no `GeneratedRepositories` type or type alias pattern.

## Adding Custom Queries and Methods

1. **Add the method signature** above the `// gopernicus:start` marker in `repository.go`:
   ```go
   type Storer interface {
       MyCustomQuery(ctx context.Context, tenantID string) ([]Invitation, error)

       // gopernicus:start
       List(ctx context.Context, ...) ([]Invitation, error)
       // ... generated methods
       // gopernicus:end
   }
   ```

2. **Implement it** in `invitationspgx/store.go`:
   ```go
   func (s *Store) MyCustomQuery(ctx context.Context, tenantID string) ([]invitations.Invitation, error) {
       rows, err := s.db.Query(ctx, `SELECT ... WHERE tenant_id = @tenant_id`, pgx.NamedArgs{"tenant_id": tenantID})
       if err != nil {
           return nil, err
       }
       defer rows.Close()
       return pgx.CollectRows(rows, pgx.RowToStructByName[invitations.Invitation])
   }
   ```

3. **Optionally add a Repository method** if the custom query needs business logic (pagination wrapping, event emission, etc.).

The custom method is automatically available through `CacheStore` because it wraps `Storer` -- uncached methods pass through to the underlying store.

Alternatively, add a new `@func` annotation to `queries.sql` in `gopernicus-cli/internal/frameworkrepos/` and re-run `gopernicus generate` to get a fully generated implementation with filters, pagination, and caching.
