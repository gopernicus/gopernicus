# Extending Generated Code

Gopernicus generates a large amount of boilerplate -- entity structs, store
interfaces, repository logic, cache wrappers, bridge handlers, and composite
wiring.  All generated files contain the header `// Code generated by
gopernicus. DO NOT EDIT.`  They are overwritten on every `gopernicus generate`
run.

Customization happens exclusively in **bootstrap files**: one-time files
created during the first generation that are never overwritten.  This document
covers each bootstrap file and the extension points they provide.

## Bootstrap files

Every repository package contains a set of bootstrap files alongside its
generated counterparts.  Using `core/repositories/rebac/invitations/` as an
example:

| File                 | Bootstrap? | Purpose                                     |
|----------------------|------------|---------------------------------------------|
| `generated.go`       | No         | Entity struct (`Invitation`), input/filter structs (`CreateInvitation`, `UpdateInvitation`, `FilterList`), errors, order-by fields (`OrderByFields`), Repository methods, cursor helpers |
| `generated_cache.go` | No         | CacheStore, CacheConfig, invalidation        |
| `repository.go`      | Yes        | Storer interface (with `gopernicus:start`/`end` markers), Repository struct, NewRepository, options |
| `fop.go`             | Yes        | Order-by field map, default sort, page size   |
| `cache.go`           | Yes        | Custom cache methods                          |

Bridge packages follow a similar pattern:

| File                 | Bootstrap? | Purpose                                     |
|----------------------|------------|---------------------------------------------|
| `generated.go`       | No         | HTTP handler methods on `*Bridge`, request types, `addGeneratedRoutes()` |
| `bridge.yml`         | Yes        | Routes, ordered middleware arrays, auth schema (`auth_relations`, `auth_permissions`) |
| `bridge.go`          | Yes        | Bridge struct (flat, with all fields), NewBridge constructor |
| `routes.go`          | Yes        | `AddHttpRoutes()` that calls `addGeneratedRoutes()` + custom routes |
| `http.go`            | Yes        | Custom HTTP handlers                          |
| `fop.go`             | Yes        | Filter/order parsing customization            |
| `subscribers.go`     | Yes        | Event subscriptions                           |

## `repository.go` -- Storer, Repository, and NewRepository

The bootstrap `repository.go` defines:

- **Storer interface**: contains generated methods between `// gopernicus:start`
  and `// gopernicus:end` markers. Custom methods go above the start marker.
- **Repository struct**: a flat struct with fields (store, generateID, bus) --
  no embedding of a generated type.
- **NewRepository**: constructor that wires dependencies.
- **Options**: functional options like `WithGenerateID` and `WithEventBus`.

```go
type Storer interface {
    // Add custom store methods above the markers:
    // GetByToken(ctx context.Context, hash string) (Invitation, error)

    // gopernicus:start
    List(ctx context.Context, filter FilterList, orderBy fop.Order, page fop.PageStringCursor, forPrevious bool) ([]Invitation, error)
    Get(ctx context.Context, invitationID string) (Invitation, error)
    Create(ctx context.Context, input CreateInvitation) (Invitation, error)
    // ... more generated methods
    // gopernicus:end
}

type Repository struct {
    store      Storer
    generateID func() string
    bus        events.Bus
}

func NewRepository(store Storer, opts ...Option) *Repository {
    r := &Repository{
        store:      store,
        generateID: cryptids.GenerateID,
    }
    for _, opt := range opts {
        opt(r)
    }
    return r
}
```

Generated methods on `*Repository` live in `generated.go`. To override a
generated repository method, define it in `repository.go` with the same
signature -- Go's method resolution will use the method on the package-level
type:

```go
// In repository.go -- overrides the generated Create in generated.go
func (r *Repository) Create(ctx context.Context, input CreateInvitation) (Invitation, error) {
    // Custom pre-processing...
    input.InvitationID = r.generateID()
    inv, err := r.store.Create(ctx, input)
    // Custom post-processing...
    return inv, err
}
```

## `fop.go` -- filtering, ordering, pagination

Controls list behavior defaults:

```go
var OrderByFields = OrderByFields
const DefaultOrderBy = OrderByCreatedAt
const DefaultOrderDirection = fop.DESC
const DefaultLimit = 25
```

Override to change default sorting, restrict orderable fields, or adjust page
size.

## `cache.go` -- custom cache methods

Created once with a comment template.  Add domain-specific invalidation or
warming logic here:

```go
func (s *CacheStore) InvalidateForTenant(ctx context.Context, tenantID string) {
    _ = s.cache.DeletePattern(ctx, s.config.KeyPrefix+":*")
}
```

## Bridge `bridge.yml` -- routes, middleware, and auth schema

The `bridge.yml` file in each bridge package directory declares routes,
ordered middleware arrays, and auth schema. Protocol annotations that
previously lived in `queries.sql` are now here:

```yaml
routes:
  - func: List
    path: /tenants/{tenant_id}/invitations
    middleware:
      - authenticate: any
      - rate_limit
      - authorize:
          pattern: prefilter
          permission: read

  - func: Get
    path: /tenants/{tenant_id}/invitations/{invitation_id}
    middleware:
      - authenticate: any
      - rate_limit
      - authorize:
          permission: read
          param: invitation_id

  - func: Create
    path: /tenants/{tenant_id}/invitations
    params_to_input:
      - tenant_id
    middleware:
      - max_body_size: 1048576
      - authenticate: any
      - rate_limit
      - authorize:
          permission: create
          param: tenant_id

auth_relations:
  - "tenant(tenant)"
  - "owner(user, service_account)"

auth_permissions:
  - "list(tenant->list)"
  - "create(tenant->manage)"
  - "read(owner|tenant->read)"
  - "manage(owner|tenant->manage)"
```

Known middleware types: `authenticate`, `authorize`, `rate_limit`,
`max_body_size`, `unique_to_id`. Custom middleware can be specified via raw Go
strings.

## Bridge `bridge.go` -- flat Bridge struct

The `Bridge` struct has ALL fields directly -- no embedding of a generated
type:

```go
type Bridge struct {
    repository    *invitations.Repository
    log           *slog.Logger
    rateLimiter   *ratelimiter.RateLimiter
    authenticator *authentication.Authenticator
    authorizer    *authorization.Authorizer
    jsonErrors    httpmid.ErrorRenderer
}
```

## Bridge `routes.go` -- route registration

`AddHttpRoutes` is defined in the bootstrap `routes.go` and receives a
`*web.RouteGroup`. It calls `addGeneratedRoutes()` (private, from
generated.go) and then registers any custom routes:

```go
func (b *Bridge) AddHttpRoutes(group *web.RouteGroup) {
    b.addGeneratedRoutes(group)
    // Custom routes:
    group.POST("/{resource_type}/{resource_id}", b.httpCreate, ...)
    group.POST("/{invitation_id}/cancel", b.httpCancel, ...)
    group.GET("/mine", b.httpListMine, ...)
}
```

## Bridge `http.go` -- custom handlers

Hand-written HTTP handlers live in `http.go`:

```go
func (b *Bridge) httpCancel(w http.ResponseWriter, r *http.Request) {
    // Custom handler logic
}
```

## Bridge `subscribers.go` -- event subscriptions

Register event handlers in the bootstrap `subscribers.go`:

```go
func (s *Subscribers) Register(bus events.Bus) error {
    sub, err := bus.Subscribe("email.verified",
        events.TypedHandler(s.handleEmailVerified))
    s.subs = append(s.subs, sub)
    return err
}
```

## Composite packages -- flat structs

Domain composites use a flat `Repositories` struct directly in
`generated_composite.go` -- no `GeneratedRepositories` type or type alias
pattern:

```go
// generated_composite.go
type Repositories struct {
    Invitation *invitations.Repository
    Group      *groups.Repository
    // ...
}

func NewRepositories(log *slog.Logger, db pgxdb.Querier, c *cache.Cache, bus events.Bus) *Repositories {
    return &Repositories{
        Invitation: invitations.NewRepository(
            invitations.NewCacheStore(invitationspgx.NewStore(log, db), c),
            invitations.WithEventBus(bus),
        ),
        // ...
    }
}
```

The same pattern applies to bridge composites (`Bridges` struct directly in
`generated_composite.go`).

## Summary of extension rules

1. **Never edit** files with the `DO NOT EDIT` header.
2. **Always extend** via bootstrap files -- they are created once and never
   overwritten.
3. Override generated behavior by defining a method with the same signature
   in the bootstrap file (e.g., `repository.go` or `http.go`).
4. Protocol and auth configuration lives in `bridge.yml`, not in `queries.sql`.
5. If generation fails for a reason it should not, flag the issue for fix in
   gopernicus rather than patching the generated output.
